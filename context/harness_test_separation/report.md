# 하네스 테스트 역할 분리 작업 최종 보고서

## 작업 요약

Generator와 Evaluator QA가 XCTest/XCUITest를 동일하게 두 번 실행하던 비효율적 구조를 개편했다. Generator는 XCTest(단위 테스트)와 XCUITest(UI 테스트) 코드를 모두 작성하되 XCTest만 직접 실행하고, XCUITest 실행은 QA가 전담하도록 `.harness/generator.md`와 `.harness/evaluator.md`를 수정했다. 이로써 각 테스트 스위트가 한 번씩만 실행되며 역할 간 책임이 명확히 분리되었다.

## 실수 및 수정 이력

| 항목 | 문제 내용 | 원인 | 수정 방법 |
|------|-----------|------|-----------|
| evaluator.md checklist.md 형식 XCUITest 행 | 템플릿의 XCUITest 행이 `\| - \| \|` 그대로 남아 Generator가 결과를 직접 기입해야 하는 것처럼 오해할 수 있었음 | 구현 시 산출물·금지 사항 수정에 집중하다 Checklist 모드의 형식 템플릿(evaluator.md:71)을 누락 | 재작업에서 `\| - (QA 담당) \| \|` 로 변경 |

## 기술 결정 배경

| 항목 | 결정 | 배경 및 이유 |
|------|------|-------------|
| XCUITest 코드 작성 주체 | Generator 유지 | 구현 맥락을 가장 잘 아는 Generator가 테스트 코드를 함께 작성해야 커버리지와 정확도가 높음 |
| XCUITest 실행 주체 | QA로 이전 | 독립적인 시각으로 검증하려면 코드 작성자와 실행자가 달라야 함. QA가 실행함으로써 Generator의 편향 없는 결과를 보장 |
| XCTest 실행 주체 | Generator 유지 | 비즈니스 로직 검증은 구현 직후 즉시 확인하는 것이 피드백 루프를 최소화함 |
| QA의 XCTest 처리 | Generator 자체 점검 결과 참조 | 결정론적(deterministic) 테스트를 재실행해도 추가 정보가 없으므로 중복 제거. QA의 독립 판단이 필요한 것은 UI 흐름이지 단위 로직이 아님 |

## 특이사항 및 다음 스프린트 권장 사항

- evaluator.md 형식 템플릿 수정 누락: 구현 범위를 산출물·금지 사항·절차에 집중하면서 Checklist 모드 내 형식 템플릿까지 점검하지 못했음. 향후 형식 템플릿이 포함된 파일 수정 시 템플릿 섹션도 체크 항목에 명시적으로 포함할 것을 권장
- Android/Espresso 테스트 역할 분리: 이번 스펙에서는 iOS(XCTest/XCUITest)만 다뤘으며 Android 측 역할 분리는 다음 스프린트에서 별도 진행 필요

## 회고

### 잘된 점
- 문제 정의(테스트 중복)에서 해결책(역할 분리)까지 논리적으로 스펙을 도출하여 구현 방향이 명확했음
- [필수] 항목 9건을 한 번의 구현 사이클에서 모두 통과시켜 재작업 범위를 최소화함
- QA에서 [권장] FAIL 1건을 정확히 포착하여 재작업→재검증 사이클이 1회로 종료됨

### 개선할 점
- 형식 템플릿처럼 "내용 파일"이 아닌 "메타 파일" 섹션까지 체크리스트 항목으로 명시적으로 포함하면 누락을 예방할 수 있음
- 구현 모드에서 수정 대상 파일의 모든 섹션을 순서대로 훑는 습관을 들이면 이 같은 누락을 줄일 수 있음
